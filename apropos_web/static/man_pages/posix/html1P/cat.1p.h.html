The Linux implementation of this interface may differ (consult the corresponding Linux manual page for details of Linux behavior), or the interface may not be implemented on Linux.</div> <div class="section"> <h1>NAME</h1> cat - concatenate and print files</div> <div class="section"> <h1>SYNOPSIS</h1> <b>cat</b> <b>[</b><b>-u</b><b>][</b><i>file</i> <b>...</b><b>]</b></div> <div class="section"> <h1>DESCRIPTION</h1> The <i>cat</i> utility shall read files in sequence and shall write their contents to the standard output in the same sequence.</div> <div class="section"> <h1>OPTIONS</h1> The <i>cat</i> utility shall conform to the Base Definitions volume of IEEE&#160;Std&#160;1003.1-2001, Section 12.2, Utility Syntax Guidelines.<div class="spacer"> </div> The following option shall be supported:<dl> <dt> <b>-u</b></dt> <dd> Write bytes from the input file to the standard output without delay as each is read.<div style="height: 1.00em;"> &#160;</div> </dd> </dl> </div> <div class="section"> <h1>OPERANDS</h1> The following operand shall be supported:<dl> <dt> <i>file</i></dt> <dd> A pathname of an input file. If no <i>file</i> operands are specified, the standard input shall be used. If a <i>file</i> is <b>'-'</b>, the <i>cat</i> utility shall read from the standard input at that point in the sequence. The <i>cat</i> utility shall not close and reopen standard input when it is referenced in this way, but shall accept multiple occurrences of <b>'-'</b> as a <i>file</i> operand.<div style="height: 1.00em;"> &#160;</div> </dd> </dl> </div> <div class="section"> <h1>STDIN</h1> The standard input shall be used only if no <i>file</i> operands are specified, or if a <i>file</i> operand is <b>'-'</b> . See the INPUT FILES section.</div> <div class="section"> <h1>INPUT FILES</h1> The input files can be any file type.</div> <div class="section"> <h1>ENVIRONMENT VARIABLES</h1> The following environment variables shall affect the execution of <i>cat</i>:<dl> <dt> <i>LANG</i></dt> <dd> Provide a default value for the internationalization variables that are unset or null. (See the Base Definitions volume of IEEE&#160;Std&#160;1003.1-2001, Section 8.2, Internationalization Variables for the precedence of internationalization variables used to determine the values of locale categories.)</dd> </dl> <dl> <dt> <i>LC_ALL</i></dt> <dd> If set to a non-empty string value, override the values of all the other internationalization variables.</dd> </dl> <dl> <dt> <i>LC_CTYPE</i></dt> <dd> Determine the locale for the interpretation of sequences of bytes of text data as characters (for example, single-byte as opposed to multi-byte characters in arguments).</dd> </dl> <dl> <dt> <i>LC_MESSAGES</i></dt> <dd> Determine the locale that should be used to affect the format and contents of diagnostic messages written to standard error.</dd> </dl> <dl> <dt> <i>NLSPATH</i></dt> <dd> Determine the location of message catalogs for the processing of <i>LC_MESSAGES</i> <i>.</i><div style="height: 1.00em;"> &#160;</div> </dd> </dl> </div> <div class="section"> <h1>ASYNCHRONOUS EVENTS</h1> Default.</div> <div class="section"> <h1>STDOUT</h1> The standard output shall contain the sequence of bytes read from the input files. Nothing else shall be written to the standard output.</div> <div class="section"> <h1>STDERR</h1> The standard error shall be used only for diagnostic messages.</div> <div class="section"> <h1>OUTPUT FILES</h1> None.</div> <div class="section"> <h1>EXTENDED DESCRIPTION</h1> None.</div> <div class="section"> <h1>EXIT STATUS</h1> The following exit values shall be returned:<dl> <dt> &#160;0</dt> <dd> All input files were output successfully.</dd> </dl> <dl> <dt> &gt;0</dt> <dd> An error occurred.<div style="height: 1.00em;"> &#160;</div> </dd> </dl> </div> <div class="section"> <h1>CONSEQUENCES OF ERRORS</h1> Default.<div class="spacer"> </div> <i>The following sections are informative.</i></div> <div class="section"> <h1>APPLICATION USAGE</h1> The <b>-u</b> option has value in prototyping non-blocking reads from FIFOs. The intent is to support the following sequence:<div style="height: 1.00em;"> &#160;</div> <div style="margin-left: 5.00ex;"> <div class="spacer"> </div> <br/> <b>mkfifo foo</b><br/> <b>cat -u foo &gt; /dev/tty13 &amp;</b><br/> <b>cat -u &gt; foo</b><br/> <b></b><br/> </div> <div class="spacer"> </div> It is unspecified whether standard output is or is not buffered in the default case. This is sometimes of interest when standard output is associated with a terminal, since buffering may delay the output. The presence of the <b>-u</b> option guarantees that unbuffered I/O is available. It is implementation-defined whether the <i>cat</i> utility buffers output if the <b>-u</b> option is not specified. Traditionally, the <b>-u</b> option is implemented using the equivalent of the <i>setvbuf</i>() function defined in the System Interfaces volume of IEEE&#160;Std&#160;1003.1-2001.</div> <div class="section"> <h1>EXAMPLES</h1> The following command:<div style="height: 1.00em;"> &#160;</div> <div style="margin-left: 5.00ex;"> <div class="spacer"> </div> <br/> <b>cat myfile</b><br/> <b></b><br/> </div> <div class="spacer"> </div> writes the contents of the file <b>myfile</b> to standard output.<div class="spacer"> </div> The following command:<div style="height: 1.00em;"> &#160;</div> <div style="margin-left: 5.00ex;"> <div class="spacer"> </div> <br/> <b>cat doc1 doc2 &gt; doc.all</b><br/> <b></b><br/> </div> <div class="spacer"> </div> concatenates the files <b>doc1</b> and <b>doc2</b> and writes the result to <b>doc.all</b>.<div class="spacer"> </div> Because of the shell language mechanism used to perform output redirection, a command such as this:<div style="height: 1.00em;"> &#160;</div> <div style="margin-left: 5.00ex;"> <div class="spacer"> </div> <br/> <b>cat doc doc.end &gt; doc</b><br/> <b></b><br/> </div> <div class="spacer"> </div> causes the original data in <b>doc</b> to be lost.<div class="spacer"> </div> The command:<div style="height: 1.00em;"> &#160;</div> <div style="margin-left: 5.00ex;"> <div class="spacer"> </div> <br/> <b>cat start - middle - end &gt; file</b><br/> <b></b><br/> </div> <div class="spacer"> </div> when standard input is a terminal, gets two arbitrary pieces of input from the terminal with a single invocation of <i>cat</i>. Note, however, that if standard input is a regular file, this would be equivalent to the command:<div style="height: 1.00em;"> &#160;</div> <div style="margin-left: 5.00ex;"> <div class="spacer"> </div> <br/> <b>cat start - middle /dev/null end &gt; file</b><br/> <b></b><br/> </div> <div class="spacer"> </div> because the entire contents of the file would be consumed by <i>cat</i> the first time <b>'-'</b> was used as a <i>file</i> operand and an end-of-file condition would be detected immediately when <b>'-'</b> was referenced the second time.</div> <div class="section"> <h1>RATIONALE</h1> Historical versions of the <i>cat</i> utility include the options <b>-e</b>, <b>-t</b>, and <b>-v</b>, which permit the ends of lines, &lt;tab&gt;s, and invisible characters, respectively, to be rendered visible in the output. The standard developers omitted these options because they provide too fine a degree of control over what is made visible, and similar output can be obtained using a command such as:<div style="height: 1.00em;"> &#160;</div> <div style="margin-left: 5.00ex;"> <div class="spacer"> </div> <br/> <b>sed -n -e 's/$/$/' -e l pathname</b><br/> <b></b><br/> </div> <div class="spacer"> </div> The <b>-s</b> option was omitted because it corresponds to different functions in BSD and System V-based systems. The BSD <b>-s</b> option to squeeze blank lines can be accomplished by the shell script shown in the following example:<div style="height: 1.00em;"> &#160;</div> <div style="margin-left: 5.00ex;"> <div class="spacer"> </div> <br/> <b>sed -n '</b><br/> <b># Write non-empty lines.</b><br/> <b>/./ {</b><br/> <b> p</b><br/> <b> d</b><br/> <b> }</b><br/> <b># Write a single empty line, then look for more empty lines.</b><br/> <b>/^$/ p</b><br/> <b># Get next line, discard the held &lt;newline&gt; (empty line),</b><br/> <b># and look for more empty lines.</b><br/> <b>:Empty</b><br/> <b>/^$/ {</b><br/> <b> N</b><br/> <b> s/.//</b><br/> <b> b Empty</b><br/> <b> }</b><br/> <b># Write the non-empty line before going back to search</b><br/> <b># for the first in a set of empty lines.</b><br/> <b> p</b><br/> <b></b><br/> </div> <div class="spacer"> </div> The System V <b>-s</b> option to silence error messages can be accomplished by redirecting the standard error. Note that the BSD documentation for <i>cat</i> uses the term &quot;blank line&quot; to mean the same as the POSIX &quot;empty line'': a line consisting only of a &lt;newline&gt;.<div class="spacer"> </div> The BSD <b>-n</b> option was omitted because similar functionality can be obtained from the <b>-n</b> option of the <i>pr</i> utility.</div> <div class="section"> <h1>FUTURE DIRECTIONS</h1> None.</div> <div class="section"> <h1>SEE ALSO</h1> <i>more</i>, the System Interfaces volume of IEEE&#160;Std&#160;1003.1-2001, <i>setvbuf</i>()</div> <div class="section"> <h1>COPYRIGHT</h1> Portions of this text are reprinted and reproduced in electronic form from IEEE Std 1003.1, 2003 Edition, Standard for Information Technology -- Portable Operating System Interface (POSIX), The Open Group Base Specifications Issue 6, Copyright (C) 2001-2003 by the Institute of Electrical and Electronics Engineers, Inc and The Open Group.
